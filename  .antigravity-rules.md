# Diretrizes de Desenvolvimento (Antigravity Rules)

Este arquivo contém as instruções permanentes que o assistente deve seguir para este projeto. Sempre leia este arquivo no início de cada interação e respeite estas regras durante análise, planejamento e execução.

## 1. Idiomas e Comunicação
- Mensagens de commit: sempre em Inglês.
- Planos de implementação (`implementation_plan.md`): sempre em Português.
- Walkthroughs e explicações: sempre em Português.
- Antes de qualquer alteração, explique de forma objetiva:
  - o que foi entendido
  - o que será feito
  - quais arquivos serão afetados
  - qual o impacto esperado

## 2. Classificação das Solicitações
### Solicitações simples
São consideradas simples:
- ajustes pontuais de texto
- correções pequenas de UI
- pequenas correções de lógica
- tradução
- sugestão de commit
- explicações sem alteração estrutural

Para solicitações simples:
- não é necessário criar `implementation_plan.md`
- a execução pode acontecer diretamente
- ainda assim, a lógica deve ser explicada antes da alteração

### Solicitações complexas
São consideradas complexas:
- criação de novas funcionalidades
- alterações em múltiplos arquivos
- refatorações
- mudanças de estrutura
- integrações com API, banco de dados ou autenticação
- mudanças que afetem UX, arquitetura ou fluxo principal

Para solicitações complexas:
1. pesquisar e analisar o contexto
2. criar `implementation_plan.md` em Português
3. solicitar aprovação
4. só depois executar

## 3. Fluxo de Trabalho
Para solicitações complexas, seguir rigorosamente esta ordem:
1. Pesquisa: analisar arquivos, contexto e impacto
2. Plano de Implementação: criar `implementation_plan.md` em Português
3. Aprovação: solicitar confirmação antes de alterar
4. Execução: após aprovação, criar `task.md` e realizar as alterações
5. Resumo: criar `walkthrough.md` em Português
6. Encerramento: fornecer sugestão de commit em Inglês ao final

## 4. Regras de Aprovação Obrigatória
Sempre pedir aprovação explícita antes de:
- alterar muitos arquivos
- modificar arquitetura
- mexer em banco de dados
- apagar arquivos
- substituir grandes blocos de código
- alterar fluxos críticos
- fazer comandos Git destrutivos

## 5. Regras de Git e Segurança
- Nunca executar comandos destrutivos sem aprovação explícita.
- Isso inclui especialmente:
  - `git reset --hard`
  - `git clean -fd`
  - `git push --force`
  - `git rebase`
  - remoção de commits ou histórico
- Sempre explicar o risco antes de sugerir qualquer comando destrutivo.
- Sempre sugerir um nome de commit ao final.
- Nunca commitar `.antigravity-rules.md` — este arquivo deve permanecer local e ignorado pelo Git.

## 6. Padrões de Implementação
- Preservar a stack atual do projeto.
- Não trocar bibliotecas sem necessidade real e justificativa clara.
- Seguir o padrão de pastas, componentes, nomenclatura e arquitetura já existente.
- Evitar duplicação de código.
- Priorizar soluções limpas, legíveis e reutilizáveis.
- Não fazer mudanças desnecessárias fora do escopo pedido.
- Sempre preservar o que já funciona, salvo quando houver motivo claro para refatorar.

## 7. Padrões Estéticos (UI/UX)
- Border radius global:
  - `rounded-lg` para cards
  - `rounded-md` para botões e inputs
- Evitar arredondamentos excessivos ou aparência amadora.
- A estética do projeto deve seguir uma linha:
  - moderna
  - minimalista
  - premium
- Referências visuais:
  - ChatGPT (OpenAI)
  - Vercel
  - Linear
- Menu de perfil:
  - estética flat
  - sem fundos coloridos nos ícones
  - cantos `rounded-xl`
  - sem animações de abertura desnecessárias

## 8. Formato Padrão de Resposta
Sempre que houver implementação, responder nesta ordem:
1. Entendimento do pedido
2. Plano resumido
3. Arquivos afetados
4. Impacto esperado
5. Execução ou solicitação de aprovação
6. Commit sugerido ao final

## 9. Qualidade e Consistência
- Não inventar comportamentos, arquivos ou dependências.
- Não afirmar que testou algo se não testou.
- Explicar limitações quando existirem.
- Manter consistência entre design, código e estrutura.
- Evitar respostas genéricas.
- Ser direto, técnico e claro.
